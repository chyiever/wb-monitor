# 2026-03-11 声发射 TCP 通信丢包问题解决记录

## 1. 文档目的

本文档记录 PCCP 断丝监测软件在 Tab1 开发早期遇到的 TCP 数据接收异常问题，以及最终形成的解决思路。  
该问题记录与当前代码结构对应关系如下：

- 当前实际实现位于 `src/fip_tab1/fip_tcp_server.py`
- Tab1 主流程由 `src/main.py` 调用
- 这份文档关注的是“为什么当前 TCP 接收方案是现在这个样子”

## 2. 问题背景

在软件开发早期，Tab1 需要持续接收 LabVIEW RT 发送的大体量 FIP 数据包。  
每个数据包的特点是：

- 传输协议：TCP
- 发送频率：约每 `0.2 s` 一包
- 头部：`8` 字节
- 包体：约 `1.6 MB`
- 数据内容：大端 `int64`，表示 `<32,32>` 定点数

在最初版本中，TCP 接收链路经常出现“看起来像丢包”的问题，导致后续预处理、绘图和统计信息异常。

## 3. 当时的主要现象

### 3.1 表现

- 丢包率统计异常，严重时可上升到很高水平
- 单包接收延迟偏大，明显超过实时处理预期
- 出现大量“旧包”“重复包”“乱序包”的告警日志
- 波形显示经常卡顿或不刷新
- 通信正常但 UI 表现为“像是没有收到数据”

### 3.2 现象本质

后续分析表明，早期问题里真正的网络传输故障并不是唯一根因。  
更大的问题来自：

- 应用层过度复杂的包序号校验
- 低效的接收实现
- 高频日志与统计造成的额外性能负担
- 对 TCP 协议特性的误用和重复防护

换句话说，当时很多“丢包”其实是应用层自己把正常包判成了异常包。

## 4. 原始实现中的关键问题

## 4.1 过度复杂的包序号校验

早期实现试图在应用层做大量额外判断：

- 严格要求 `comm_count` 必须严格递增
- 尝试检测重复包
- 尝试检测乱序包
- 一旦不符合预期就直接丢弃

这种策略在理想情况下看起来合理，但在实际工程里会带来几个问题：

- 一旦发送端重连或计数器复位，接收端会把后续正常包误判为异常
- 应用层重复实现了 TCP 已经提供的可靠传输保证
- 错误处理逻辑越复杂，越容易把“边界情况”误当成“错误”

## 4.2 接收逻辑效率低

早期版本的接收方式偏向“小块、频繁、重复拼接”，会带来：

- 系统调用次数过多
- 大包接收时间明显上升
- 高峰时更容易造成后续处理线程积压

## 4.3 日志和监控过密

在调试阶段加入的大量逐包日志，在运行阶段反而成了性能负担：

- 每包都打印详细调试信息
- 每包都计算大量统计指标
- 日志 I/O 放大了接收线程负担

结果是：

- CPU 时间被日志和诊断吞掉
- 真正的数据接收吞吐下降
- 调试行为本身又进一步放大了问题

## 4.4 协议解析理解不够稳定

早期版本在协议细节上经历过几轮修正，主要包括：

- 头部长度理解不稳定
- 包体长度和点数换算容易混淆
- 定点数转换逻辑需要明确统一

这些问题会直接放大误判和异常日志。

## 5. 解决思路

最终解决该问题的方向，不是继续堆更多“补丁式校验”，而是回到更稳定的工程原则：

- 简化协议处理链路
- 相信 TCP 的点对点可靠传输特性
- 把接收路径做成“少判断、少阻塞、少日志”
- 只保留真正必要的边界检查

## 6. 核心改进项

## 6.1 统一为 8 字节头部协议

当前接收逻辑固定为：

- 前 `4` 字节：`raw_comm_count`
- 后 `4` 字节：`data_length`

解析方式：

```python
raw_comm_count, data_length = struct.unpack(">II", header_bytes)
```

这让协议入口变得明确，减少了协议理解偏差。

## 6.2 使用 `_recv_exact()` 完整收包

当前实现采用“必须收满指定字节数再继续”的方式：

- 先收满头部
- 再收满包体
- 半包情况下继续循环，不提前误判

并且使用 `64 KB` 分块接收：

```python
chunk_size = min(bytes_needed, 65536)
chunk = self.client_socket.recv(chunk_size)
```

这样做的优点是：

- 降低系统调用频率
- 对大包更友好
- 行为更可预测

## 6.3 放弃应用层重复包/乱序包强校验

当前 `OptimizedTCPServer` 不再做复杂的应用层重复包过滤。  
只保留：

- 基础长度校验
- 连接状态管理
- 会话内通信计数归一化

这避免了把“发送端重置”和“会话切换”误判成异常包。

## 6.4 保留通信计数归一化，但降低侵入性

当前保留 `raw_comm_count -> comm_count` 的归一化逻辑：

- 新连接的第一包强制归一化为 `0`
- 若发现原始计数显著回退，则重新建立基准

这样做的目的不是为了“拒绝包”，而是为了：

- 让上层 UI 和处理线程拥有稳定的会话内编号
- 让波形线程更容易识别“重连/复位”场景

## 6.5 调整日志频率

当前策略是：

- 关键事件打印日志
- 性能统计按周期打印
- 不再对每个包做高密度调试输出

例如，性能统计按每 `50` 包记录一次。

## 6.6 加强 TCP 参数配置

当前实现里显式做了以下优化：

- `TCP_NODELAY = 1`
- `SO_RCVBUF = 8MB`

目的：

- 降低传输时延
- 增大接收缓冲区
- 降低高吞吐下的拥塞风险

## 7. 当前实现对应关系

这些优化最终落实在当前文件：

- `src/fip_tab1/fip_tcp_server.py`

当前该模块负责：

- 建立 TCP 服务端
- 接收和解析数据包
- 将 `<32,32>` 定点数转为 `float64`
- 发出 `DataPacket`
- 记录性能统计
- 处理连接断开与会话切换

主程序中由：

- `src/main.py`

完成接收后信号连接：

- `self.tcp_server.data_received.connect(self._process_data_packet)`

也就是说，当前 Tab1 的整个后续处理链都建立在这次 TCP 接收重构结果之上。

## 8. 当前协议说明

### 8.1 数据格式

| 项目 | 内容 |
|---|---|
| 头部格式 | `>II` |
| 头部长度 | `8` 字节 |
| 头部字段 1 | `raw_comm_count` |
| 头部字段 2 | `data_length` |
| 包体格式 | `>q` 重复 `N` 次 |
| 单点类型 | 大端 `int64` |
| 数值语义 | `<32,32>` 定点数 |
| 转换方式 | `float_value = int64_value / 2**32` |

### 8.2 默认传输特征

| 项目 | 值 |
|---|---|
| 原始采样率 | `1 MHz` |
| 每包点数 | `200000` |
| 包体长度 | `1600000` 字节 |
| 发送周期 | `0.2 s` |

## 9. 问题解决后的工程收益

这次 TCP 接收链路优化带来的收益，不只是“丢包率下降”，更重要的是为整个 Tab1 主链路提供了稳定输入：

- Tab1 后台线程能持续收到完整包
- 时域绘图不再因异常判包而中断
- PSD 计算输入更稳定
- Tab2 得到的下游数据也更连续

所以，这次修改本质上是整个软件架构稳定化的第一步。

## 10. 经验总结

### 10.1 不要在 TCP 之上重复实现一整套“可靠传输”

对于当前这种点对点、直连、固定协议的大包场景，应用层更应该关注：

- 包是否完整
- 协议是否一致
- 会话状态是否可控

而不是把大量复杂的“丢包检测”“乱序修复”堆在接收层。

### 10.2 高吞吐链路优先保证主路径简单

实时系统中，最怕的是接收路径上堆太多判断。  
接收层越简单，上层越稳定。

### 10.3 调试信息不能破坏实时链路

调试日志必须节流。  
否则“为了看问题”会反过来制造问题。

### 10.4 成功方案应当沉淀为结构化模块

早期问题解决后，这套 TCP 逻辑没有停留在临时脚本层，而是最终沉淀为：

- `src/fip_tab1/fip_tcp_server.py`

这也是后续重构 Tab1 目录时，将 TCP 接收逻辑纳入 `fip_tab1` 独立包的原因之一。

## 11. 当前结论

截至 2026-03-13，这次 TCP 通信问题已经不再是当前系统的主要瓶颈。  
当前实现说明：

- TCP 主链路已稳定可用
- Tab1 和 Tab2 的后续功能建立在这套接收实现之上
- 当初的“高丢包率”问题已经通过协议简化、接收优化和日志节流得到工程化解决

这份问题记录保留的价值，在于解释当前 `fip_tab1/fip_tcp_server.py` 的设计取舍，而不仅仅是记录一次历史故障。
