# 2026-03-20 Tab1 干涉仪数据存储技术说明

## 1. 文档目的

本文档整理 `Tab1 / FIP` 当前数据存储链路的实际实现细节。结论以代码为准，主要覆盖：

- 数据是否经过预处理
- 存储数据的采样率和长度
- 存储路径与文件名规则
- 缓存与队列机制
- 对 UI 流畅性和卡顿风险的影响

相关代码位置：

- `src/fip_tab1/fip_tcp_server.py`
- `src/fip_tab1/fip_tab1_manager.py`
- `src/processing/phase_unwrap.py`
- `src/processing/signal_filter.py`
- `src/processing/downsampling.py`

## 2. 总体结论

Tab1 当前存储的不是原始 TCP 负载，也不是滤波后数据，而是：

1. TCP 定点数转 `float64`
2. 相位展开
3. 仅在存储分支再做 `5x` 降采样
4. 以 `np.savez_compressed` 写入 `.npz`

因此，当前存储文件中的 `phase_data` 含义是：

- 相位展开后数据
- 仅用于存储的 `5x` 降采样结果
- 采样率 `200 kHz`

其他功能链路不受这个存储专用降采样影响：

- 时域绘图仍使用主处理链路的 `downsampled_data`
- PSD 仍使用 `psd_data`
- 其他处理逻辑采样率不因存储而改变

## 3. 数据接收与预处理

### 3.1 TCP 原始数据格式

Tab1 从 LabVIEW 侧接收 TCP 数据包：

- 8 字节头部：`>II`
- 包含 `raw_comm_count` 和 `data_length`
- 负载按大端 `int64` 解析：`>{point_count}q`

定点数转换公式：

```python
data_array = np.array(raw_values, dtype=np.float64) / (2**32)
```

默认每包数据：

- 数据字节数：`1600000`
- 原始采样点数：`200000`
- 原始等效采样率：`1 MHz`
- 单包时长：`0.2 s`

### 3.2 存储前实际经过的预处理

存储分支的数据来自 `ProcessedData.unwrapped_data`，因此在落盘前已经过：

1. `int64 -> float64`
2. 必要时的量纲兼容处理
3. 相位展开

代码中的兼容逻辑：

```python
phase_data = packet.phase_data
if np.max(np.abs(phase_data)) > 5:
    phase_data = phase_data / np.pi
```

这段逻辑的设计意图是兼容不同上游量纲：

- 如果上游已经送来近似 `[-pi, pi]` 量纲数据，则先除以 `pi`
- 再交给 `PhaseUnwrapper`
- `PhaseUnwrapper` 假设输入是 `[-1, 1]` 量纲，并内部乘 `pi` 后执行 `np.unwrap`

如果上游稳定输出 `[-1, 1]` 数据，则这段分支不会触发，不会影响结果。

### 3.3 不会写入存储的数据

以下数据不会直接写入 Tab1 存储文件：

- 原始 `phase_data`
- 滤波后的 `filtered_data`
- 时域显示专用 `display_data`
- PSD 计算结果

## 4. 当前存储数据的采样率与长度

### 4.1 当前规则

当前实现已经改为：

- 相位展开后，进入存储前执行 `5x` 降采样
- 使用切片方式：`unwrapped_data[::5]`

对应采样率变化：

- 相位展开前后主链路采样率：`1 MHz`
- 仅存储分支采样率：`200 kHz`

### 4.2 每个文件的数据长度

在默认输入条件下：

- 原始单包点数：`200000`
- 存储前 `5x` 降采样后点数：约 `40000`

因此当前单个存储文件中的 `phase_data` 长度通常为：

- `40000` 点

### 4.3 文件内记录的采样率

`.npz` 文件中的 `sample_rate` 字段当前写入：

- `200000.0`

同时 `data_info` 中额外记录：

- `type`
- `length`
- `downsample_factor`
- `save_time`

## 5. 存储文件格式

Tab1 使用压缩的 NPZ 文件：

```python
np.savez_compressed(...)
```

当前写入字段如下：

- `phase_data`
- `comm_count`
- `timestamp`
- `sample_rate`
- `data_info`

字段含义：

- `phase_data`: 相位展开后、仅供存储的 `5x` 降采样数据
- `comm_count`: 当前会话内归一化后的包计数
- `timestamp`: 当前实现中沿用上游包时间戳字段
- `sample_rate`: `200000.0`
- `data_info.length`: 当前数组长度，通常约 `40000`

## 6. 存储路径

Tab1 存储线程默认路径为：

```text
D:/PCCP/FIPdata
```

同时 UI 中的默认路径也是：

```text
D:/PCCP/FIPdata
```

运行时实际使用的路径来自 UI 设置：

- 勾选存储开关后
- 主程序通过 `update_storage_path()` 同步到 `DataStorageThread`

目录不存在时会自动创建：

```python
base_path.mkdir(parents=True, exist_ok=True)
```

## 7. 文件名定义

文件名规则为：

```text
phase_data_YYYYMMDD_HHMMSS_mmm_#xxxxxx.npz
```

示例结构：

```text
phase_data_20260320_091953_489_#000200.npz
```

其中：

- `YYYYMMDD`: 日期
- `HHMMSS_mmm`: 时分秒和毫秒
- `#xxxxxx`: `comm_count`，6 位零填充

## 8. 存储频率

当前存储线程内部仍然采用固定包计数节流：

- `storage_interval = 10`
- 即每收到 `10` 个存储请求，实际写盘 `1` 次

注意：

- UI 上虽然存在“存储间隔(s)”输入框
- 但当前真实实现仍是按“包数”节流，不是按秒节流
- 因此该 UI 项和当前代码实现并不完全一致

## 9. 缓存、队列与线程设计

### 9.1 线程分工

Tab1 主链路采用多线程设计：

- TCP 接收
- 数据处理线程
- 时域绘图线程
- PSD 线程
- 存储线程

存储在独立线程中完成，不阻塞 UI 主线程。

### 9.2 相关队列

当前关键队列大小如下：

- 数据处理队列：`Queue(maxsize=20)`
- 时域绘图队列：`Queue(maxsize=10)`
- PSD 队列：`Queue(maxsize=5)`
- 存储队列：`Queue(maxsize=50)`

TCP 侧还设置了较大的 socket 接收缓冲：

- `SO_RCVBUF = 8MB`

### 9.3 队列满时的策略

系统总体采用“限长队列 + 非阻塞 + 允许丢弃”的策略：

- 数据处理队列满时，优先丢最旧包
- 绘图队列满时，直接丢包
- 存储队列满时，直接丢存储请求

这说明设计目标是：

- 优先保证主链路持续运行
- 避免因为磁盘写入或绘图造成 UI 卡死
- 高负载下宁可丢部分显示或存储，也不阻塞接收线程

## 10. 会不会卡

从当前设计看，Tab1 的存储功能不太容易直接把界面卡死，原因是：

1. 存储在线程中独立执行
2. 存储队列有上限，不会无限积压
3. 队列操作是非阻塞的
4. 只按间隔写入，不是每包都同步落盘

但这并不表示完全没有风险。

高负载下更可能出现的是：

- 存储请求被丢弃
- 部分绘图包被丢弃
- 数据处理队列丢弃旧包

也就是说，当前设计更偏向“实时性优先”，而不是“每包严格可靠存储优先”。

## 11. 当前存储相关已知注意事项

### 11.1 存储数据已经不是原始 1 MHz 数据

当前存储文件中的 `phase_data` 是：

- 相位展开后
- 存储前额外 `5x` 降采样

因此如果后续离线分析需要恢复最原始的 `1 MHz` 相位展开结果，当前存储文件已不再提供该完整分辨率数据。

### 11.2 存储专用降采样只作用于存储分支

本次规则明确限定：

- 只有存储分支改为 `200 kHz`
- 其他功能数据采样率与主链路行为保持原状

这是当前实现中的重要边界条件。

### 11.3 UI 的存储间隔与代码仍未完全打通

当前实际写盘节流仍由：

```python
self.storage_interval = 10
```

控制，而不是直接使用 UI 中的“秒级间隔”配置。

## 12. 汇总

Tab1 当前存储链路可以概括为：

```text
TCP int64 定点数据
-> float64
-> 相位展开
-> 仅存储分支 5x 降采样
-> 200 kHz NPZ 压缩存储
```

核心参数汇总：

- 原始采样率：`1 MHz`
- 原始每包点数：`200000`
- 存储前降采样：`5x`
- 存储采样率：`200 kHz`
- 当前单文件长度：约 `40000` 点
- 默认存储路径：`D:/PCCP/FIPdata`
- 文件格式：`.npz`
- 文件名格式：`phase_data_YYYYMMDD_HHMMSS_mmm_#xxxxxx.npz`
- 存储线程队列：`50`
- 当前存储节流：每 `10` 个请求保存 `1` 次
