# 2026-03-12 Tab2 干涉仪信号短时特征提取与异常检测开发日志（初步）

## 1. 功能概述

本次开发完成了 FIP 干涉仪 Tab2 的第一版独立分析子系统，目标是在不破坏 Tab1 已有通信、预处理、波形显示架构的前提下，实现干涉仪短时特征提取、阈值检测、特征曲线绘图、异常事件聚合和触发存储。

Tab2 当前面向的业务能力如下：

- 对 Tab1 输出的系统滤波、降采样后的干涉仪数据进行二次带通预处理。
- 以滑窗方式连续计算短时特征，默认窗长 0.2 s、重叠率 50%。
- 按前面板勾选项分别控制“是否计算”和“是否绘图”。
- 对每个短时窗内的多个特征分别执行阈值检测。
- 将同一时间窗内的多特征检测结果聚合为异常结论，再将连续异常窗聚合为一个报警事件。
- 将报警事件前后设定时长内的时域数据和特征序列存储为 `.npz` 文件，并额外写入详细 CSV 表。

## 2. 整体架构、线程和数据流设计

### 2.1 Tab2 与 Tab1 的关系

Tab2 设计为独立子系统，不参与 Tab1 的通信、时域图、PSD 图和存储线程调度。Tab1 只在 `ProcessedData` 产出后，将其中的 `downsampled_data`、`effective_rate`、`comm_count` 转发给 Tab2。该设计的目的有两个：

- 最大限度复用 Tab1 已验证的通信与预处理链路。
- 将 Tab2 对实时分析的扩展影响限制在最小范围，降低对 Tab1 稳定性的干扰。

当前逻辑关系如下：

`TCP -> Tab1 DataProcessingThread -> ProcessedData -> {Tab1 Time Plot, Tab1 PSD Plot, Tab1 Storage, Tab2 Manager}`

### 2.2 Tab2 线程划分

Tab2 当前采用独立 QThread 工作线程，每个线程负责单一职责。

1. `FIPFeatureWorker`
   - 输入：Tab1 输出的 `downsampled_data`。
   - 功能：Tab2 专用带通滤波、滑窗拼接、短时特征计算。
   - 输出：`FIPFeatureFrame` 和用于触发存储的滤波后时域包。

2. `FIPDetectionWorker`
   - 输入：`FIPFeatureFrame`。
   - 功能：按特征分别进行阈值检测，维护基线，聚合异常窗为报警事件。
   - 输出：单窗检测结果、报警事件、触发存储请求。

3. `FIPFeaturePlotWorker`
   - 输入：特征帧和检测结果。
   - 功能：维护短时特征历史缓存，生成前面板绘图数据。
   - 输出：最多 4 路特征绘图载荷。

4. `FIPTriggerStorageWorker`
   - 输入：滤波后时域包、特征帧、触发存储请求。
   - 功能：维护环形缓存，按事件截取前后窗数据，写 `.npz` 与 `alarm_details.csv`。

5. `FIPTab2Manager`
   - 功能：连接 Tab2 各线程、同步 UI 参数、接收 Tab1 转发数据。

### 2.3 数据流设计

Tab2 的数据流分为 3 条并行子链路：

1. 分析链路
   - `ProcessedData.downsampled_data -> FIPFeatureWorker -> FIPFeatureFrame -> FIPDetectionWorker`

2. 绘图链路
   - `FIPFeatureFrame + FIPWindowDetectionResult -> FIPFeaturePlotWorker -> UI`

3. 存储链路
   - `Filtered packet + FIPFeatureFrame + FIPTriggerSaveRequest -> FIPTriggerStorageWorker -> NPZ/CSV`

### 2.4 时间轴设计

Tab2 不依赖电脑当前时间驱动业务逻辑，而是采用“样本序号驱动的相对时间轴”。

- 输入包时间：根据累计样本数和采样率换算得到。
- 窗时间：`window_index * hop_seconds`。
- 事件时间：由异常窗的开始/结束时间得到。
- 存储时间：依据事件时间和前后触发时长计算。

该设计可以保证特征时间、异常时间、存储时间使用统一时间基准，避免因系统时钟或包间抖动造成时间错位。

## 3. 当前已集成的短时特征与异常检测原理

### 3.1 已集成短时特征

当前 Tab2 已集成以下 FIP 短时特征：

- `short_energy`：短时能量，定义为滑窗内样本平方均值。
- `zero_crossing`：过零率，定义为滑窗内符号变化占比。
- `peak_factor`：峰值因子，定义为峰值绝对值与 RMS 的比值。
- `rms`：均方根值。

默认启用计算的特征为 `short_energy`，其余特征保留为可勾选扩展项。

### 3.2 异常检测与报警原理

当前异常检测采用“单特征阈值检测 + 多窗事件聚合”的方式：

1. `FIPFeatureWorker` 对每个滑窗输出多个瞬时特征值。
2. `FIPDetectionWorker` 为每个特征维护一条基线历史。
3. 阈值按 `baseline * factor` 计算；当基线极小或尚未建立时，使用阈值系数本身作为保护阈值。
4. 某一特征在某一窗内超过阈值，则该特征在该窗判定为异常。
5. 某一窗内若至少有一个特征异常，则该窗判定为异常窗。
6. 连续异常窗聚合为一个报警事件，输出事件起止时间、持续时间、触发特征名称集合和触发特征数目。

### 3.3 当前报警输出内容

前面板报警表格输出 3 列：

- `Time`
- `Duration`
- `Feature Count`

本地存储输出两类文件：

1. `an-yyyymmdd-hhmmss.xxx.npz`
   - `signal`
   - `signal_time`
   - `sample_rate`
   - `event_start_time`
   - `event_end_time`
   - `event_duration`
   - `trigger_feature_names`
   - `trigger_feature_count`
   - `feature_times`
   - `feature_values`

2. `alarm_details.csv`
   - 逐事件、逐特征、逐异常窗记录详细值。
   - 包含特征值、基线、阈值、是否触发等信息。

## 4. 本次开发涉及的主要程序文件及功能

### 4.1 Tab2 独立模块

- `src/fip_tab2/fip_types.py`
  - 定义 Tab2 公共数据结构，例如输入包、特征帧、检测结果、报警事件、触发请求。

- `src/fip_tab2/fip_feature_worker.py`
  - 负责 Tab2 带通滤波、连续缓冲、滑窗切片、短时特征计算。

- `src/fip_tab2/fip_detection_worker.py`
  - 负责特征基线维护、阈值检测、异常窗聚合和报警事件输出。

- `src/fip_tab2/fip_plot_worker.py`
  - 负责短时特征历史缓存与最近 60 s 特征曲线绘图数据生成。

- `src/fip_tab2/fip_trigger_storage.py`
  - 负责环形缓存、触发前后窗截取、报警 `.npz` 和详细 CSV 存储。

- `src/fip_tab2/fip_tab2_manager.py`
  - 负责线程连接、UI 参数同步、接收 Tab1 分发数据。

### 4.2 集成相关文件

- `src/main.py`
  - 仅做最小集成：初始化 `FIPTab2Manager`，连接 Tab1 输出到 Tab2 输入，同步前面板参数。

- `src/ui/main_window.py`
  - 增加 Tab2 参数区、双勾选框、4 路特征图、报警表格和相关 UI 接口。

### 4.3 读取工具

- `read/read_fip_alarm_npz.py`
  - 作为离线诊断工具读取 Tab2 输出的告警 `.npz` 文件，便于核查存储内容。

## 5. 更换或扩展短时特征的维护指南

### 5.1 更换特征的基本步骤

当需要将当前短时特征替换为新的特征集合时，建议按照以下顺序修改：

1. 修改 `src/fip_tab2/fip_types.py`
   - 更新 `FEATURE_DISPLAY_NAMES`，保证新特征在 UI 中有可读名称。

2. 修改 `src/fip_tab2/fip_feature_worker.py`
   - 在 `compute_enabled` 默认表中增加或移除特征键。
   - 在 `_compute_feature_values()` 中实现对应特征公式。

3. 修改 `src/fip_tab2/fip_detection_worker.py`
   - 在 `threshold_factors`、`baseline_histories`、`current_baselines` 中同步新增或删除特征。

4. 修改 `src/fip_tab2/fip_plot_worker.py`
   - 在 `plot_enabled` 和 `series` 中同步维护特征键，保证可绘图。

5. 修改 `src/ui/main_window.py`
   - 为新特征增加“Compute / Plot”勾选框、阈值输入框、基线显示项。

6. 修改 `read/read_fip_alarm_npz.py`
   - 若需要新的离线展示格式，可补充输出逻辑。

### 5.2 维护原则

建议未来维护时遵循以下原则：

- 单一数据键原则：一个特征在所有模块中保持完全一致的字符串键名。
- 单向数据流原则：特征计算、检测、绘图、存储不要交叉写逻辑。
- 最小集成原则：新增特征优先在 `src/fip_tab2` 内完成，不扩散到 Tab1 线程。
- 统一时间轴原则：所有新特征必须沿用样本序号驱动的相对时间轴。
- 先离线验证再实时联调：先用固定数据验证公式和阈值，再接入实时链路。

### 5.3 推荐扩展方式

若后续特征数量明显增加，建议进一步做两项结构化优化：

1. 将特征定义抽象成注册表
   - 将“键名 / 显示名 / 计算函数 / 默认阈值 / 默认启用状态”统一收敛到一处。

2. 将检测策略抽象成独立配置
   - 允许不同特征使用不同的阈值策略、基线窗口长度和事件判定规则。

## 6. 当前已知限制

本次开发属于初步版本，当前仍有以下已知限制：

- 报警检测仍以简单倍数阈值为主，尚未引入更复杂的自适应统计模型。
- 特征集合目前仍是硬编码配置，尚未完全注册化。
- 与发送端通信计数重复包的问题无直接关系，Tab2 只消费 Tab1 已处理输出。
- 读取脚本为运维辅助工具，不属于主程序实时链路。

## 7. 本次开发结论

本次开发完成了 Tab2 的第一版工程化落地，实现了从 Tab1 输出数据到 Tab2 特征提取、检测、绘图和触发存储的闭环。Tab1 主架构未被改写，Tab2 作为独立分析子系统可继续迭代。后续工作重点应放在：

- 丰富短时特征集合与注册机制。
- 优化基线和阈值检测策略。
- 提升报警事件判定的鲁棒性。
- 完善离线回放与验证工具链。
