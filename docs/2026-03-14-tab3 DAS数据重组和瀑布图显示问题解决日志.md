# 2026-3-14 Tab3 DAS数据重组和瀑布图显示问题解决日志

## 背景

在对 `Tab3 DAS` 功能进行联调测试时，发现：

- DAS 一维曲线显示正常
- `space-time` 瀑布图显示异常
- 日志中出现 `ImageItem` 渲染异常和缩放接口调用异常

相关异常日志如下：

```text
Exception: levels argument is required for float input types
TypeError: scale(self): too many arguments
```

这说明问题主要集中在 `Tab3` 的二维图像显示链路，而不是 TCP 收包或一维曲线数据提取。

## 问题梳理

### 1. 瀑布图几何映射方式错误

#### 现象

日志中出现：

```text
TypeError: scale(self): too many arguments
```

对应位置为 `main_window.py` 中对 `ImageItem` 的更新逻辑：

- 先 `resetTransform()`
- 再 `setPos(x_offset, y_offset)`
- 再调用 `scale(x_scale, y_scale)`

#### 原因

当前环境中的 `pyqtgraph 0.13.3` / `PyQt5` 下，`ImageItem.scale()` 不是这里预期的二维缩放接口，传入两个参数会直接抛出 `TypeError`。

因此，瀑布图虽然收到了矩阵数据，但无法把图像正确映射到时间轴和通道轴上。

#### 解决方案

改为使用 `ImageItem.setRect(x, y, width, height)` 一次性设置图像覆盖范围：

- `x` 对应时间轴起点
- `y` 对应通道起点
- `width` 对应整幅图像在时间轴上的跨度
- `height` 对应整幅图像在通道轴上的跨度

这样可以稳定完成平移和缩放映射，不再依赖错误的 `scale(x, y)` 调用。

## 2. 瀑布图矩阵轴顺序与 ImageItem 默认解释不一致

### 现象

即使一维曲线正确，瀑布图仍然可能表现为：

- 横纵方向颠倒
- 时间轴和通道轴不对应
- 图像被错误拉伸或方向异常

### 原因

`DASPlotWorker` 输出的 `space_time_matrix` 是标准的二维结构：

- 第 0 维是 `channel`
- 第 1 维是 `time`

也就是矩阵逻辑为：

```text
[channel, time]
```

而 `pyqtgraph.ImageItem` 默认兼容旧行为，二维图像通常按 `col-major` 解释。如果不显式指定轴顺序，就容易导致显示方向与业务语义不一致。

### 解决方案

在 `Tab3` 初始化 `ImageItem` 时显式指定：

```python
pg.ImageItem(axisOrder="row-major")
```

这样二维矩阵会按“行对应 Y，列对应 X”的方式显示，与当前业务数据结构匹配：

- 行方向为通道
- 列方向为时间

从而保证 `space_time_matrix[channel, time]` 能正确映射到：

- X 轴：时间
- Y 轴：通道

## 3. 浮点图像缺少稳定的 levels，导致渲染不稳定

### 现象

日志中出现：

```text
Exception: levels argument is required for float input types
```

这说明 `ImageItem` 在渲染浮点矩阵时，没有拿到可靠的显示范围。

### 原因

`space-time` 瀑布图使用的是 `float64` 矩阵。对于浮点图像，`pyqtgraph` 渲染时需要明确的亮度范围 `levels=[min, max]`。

如果仅依赖 `autoLevels=True`，在某些边界时刻可能出现：

- 内部状态切换不稳定
- 空图和正常图之间切换时状态不一致
- 渲染线程拿到的 levels 不可靠

最终触发浮点图像渲染异常。

### 解决方案

对瀑布图矩阵显式计算并传入 `levels`，不再完全依赖自动推断。

当前采用的策略为：

- 优先取有限值的 `1%` 和 `99%` 分位数作为显示范围
- 如果分位数无效，再退回到全局 `min/max`
- 如果矩阵退化为常数矩阵，则给出一个最小可显示范围

这样做的好处：

- 浮点图像渲染更稳定
- 对偶发尖峰值不那么敏感
- 避免瀑布图一会儿过曝、一会儿全黑

## 4. 空图重置路径与正常显示路径不统一

### 现象

在无数据、停止监测或重置界面时，`space-time` 图会被置为一个 `1x1` 的空矩阵。若空图状态没有统一设置 `levels` 和显示范围，容易造成后续恢复显示时状态残留。

### 原因

空图路径和正常图路径之前使用的是两套逻辑：

- 空图直接 `setImage(np.zeros((1, 1)), autoLevels=False)`
- 正常图使用另一套平移/缩放逻辑

这会导致 `ImageItem` 的内部显示状态不一致。

### 解决方案

增加统一的空图重置方法，固定：

- 空矩阵 dtype
- 空图 levels
- 空图 rect

保证每次重置后，瀑布图都回到确定状态，再接收下一帧正常数据。

## 实施顺序

本次修复按以下顺序进行：

1. 修复 `ImageItem` 的几何映射方式，去掉错误的 `scale(x_scale, y_scale)`
2. 显式指定 `ImageItem(axisOrder="row-major")`
3. 为浮点瀑布图显式计算并设置 `levels`
4. 统一空图重置逻辑

## 修复结果

本次修复后，`Tab3 DAS` 的 `space-time` 显示链路具备以下改进：

- 图像不会再因为 `scale(self): too many arguments` 报错
- 瀑布图矩阵的时间轴和通道轴映射关系明确
- 浮点图像拥有稳定的显示范围，不再依赖不可靠的自动状态
- 空图与正常图之间切换更加一致

## 后续建议

为了进一步验证修复效果，建议在联调中重点检查以下几点：

1. 瀑布图横轴是否随时间连续推进
2. 瀑布图纵轴是否对应实际通道编号
3. 修改 `channel_start/channel_end/time_downsample/space_downsample` 后，图像范围是否同步变化
4. 启停监测、多次重连后，瀑布图是否还能稳定恢复
5. 极低幅值、常数输入、含少量异常尖峰输入时，图像对比度是否仍然可用
