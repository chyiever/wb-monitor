# PCCP 断丝监测软件 Tab3/Tab4 详细设计

日期：2026-03-13

## 1. 目标与边界

本文档用于固化当前关于 `tab3` 和 `tab4` 的设计结论，作为后续编码和联调的直接依据。

### 1.1 Tab3 职责

`tab3` 只负责以下内容：

- DAS TCP 服务端通信
- DAS 数据包接收、解包、校验、二维矩阵恢复
- DAS 与 FIP 按 `comm_count` 对齐
- 实时波形显示
- 曲线 1 / 曲线 2 显示
- DAS `space-time` 图显示
- 对齐后原始联合数据存储
- 对齐状态、缺失包状态、连接状态显示

`tab3` 不负责以下内容：

- DAS 特征提取结果展示
- 事件触发定位
- 触发片段分析
- 触发结果存储

### 1.2 Tab4 职责

`tab4` 只负责以下内容：

- DAS 多特征提取
- 位置-特征图显示
- 接收 `tab2` 的 FIP 异常事件
- 按触发事件提取对齐后的 DAS/FIP 数据窗口
- 事件定位
- 触发存储
- 结果 CSV 输出

### 1.3 现有模块关系

- `tab1`：FIP 接收、预处理、时域图、PSD、实时原始存储
- `tab2`：FIP 特征提取、异常检测、告警事件输出
- `tab3`：DAS 接收、对齐、实时显示、联合原始存储
- `tab4`：定位分析与结果输出

## 2. 关键业务规则

### 2.1 会话定义

- 用户点击一次“开始监测”到下一次“停止监测”为一个采集会话。
- 每个采集会话内，FIP 与 DAS 的 `comm_count` 均从 `0` 开始。
- 用户再次点击“开始监测”时，新的采集会话重新编号。

### 2.2 对齐规则

- 对齐不依赖系统桌面时间。
- 对齐主键为 `comm_count`。
- 包内时间轴按本包采样率和样本索引推导。
- 统一时间轴表达式：
  - `packet_start_time = comm_count * packet_duration_seconds`
  - `sample_time = packet_start_time + sample_index / sample_rate`

### 2.3 通信异常处理

- 跳号、丢包：不终止当前会话。
- 缺失包在对齐缓存中标记为缺失区间。
- 定位或导出时，如时间窗内存在缺失包，则按补零策略继续处理。
- 结果中必须记录：
  - 缺失包范围
  - 缺失包比例
  - 数据完整性状态
- 若某一侧持续 `10s` 未收到新数据：
  - 立即弹窗提示用户
  - 将该侧标记为断连
  - 未断连一侧继续正常接收、显示、处理和存储
- 若某一侧在未点击“停止监测”的情况下重新从 `comm_count = 0` 开始：
  - 视为设备侧会话重置
  - 当前联合对齐状态标记为失效
  - 需要用户重新开始监测以恢复双源对齐

### 2.4 补零规则

- 当对齐时间窗内某些 FIP 或 DAS 包缺失时，缺失片段补零。
- 补零只用于保持矩阵/波形长度一致，不代表真实采样值。
- 所有补零行为都必须在结果和日志中显式记录。

### 2.5 距离映射规则

- DAS 内部通道号从 `0` 开始。
- 第 `0` 通道对应距离不一定为 `0m`。
- 前面板参数：
  - `start_distance_m`
  - `channel_spacing_m`
- 距离计算公式：
  - `distance_m = start_distance_m + channel_index * channel_spacing_m`

## 3. DAS 协议设计

### 3.1 包头定义

大端序。

包头顺序如下：

1. `comm_count: uint32`
2. `sample_rate_hz: uint32`
3. `channel_count: uint32`
4. `data_bytes: uint32`
5. `packet_duration_seconds: float64`

包体：

- `data_payload: float64[]`

### 3.2 包体验证

验证规则：

- `channel_count >= 1`
- `data_bytes > 0`
- `data_bytes % 8 == 0`
- `sample_rate_hz > 0`
- `packet_duration_seconds > 0`
- `payload_float_count == data_bytes / 8`
- `payload_float_count == sample_rate_hz * packet_duration_seconds * channel_count`

不满足时：

- 丢弃当前包
- 记录错误日志
- 更新 UI 错误状态

### 3.3 通道数量范围拆分

- 协议支持最大通道数：`44000`
- 典型联调通道数：`200`
- 当前软件前面板允许的可配置范围：`1 ~ 4000`

说明：

- 大于 `4000` 的场景保留协议兼容性，但本期默认不作为 UI 常规使用范围。
- 若后续需要支持高通道数，应重点评估 `space-time` 图和内存占用。

## 5. 数据流设计

### 5.1 总体数据流

1. FIP client -> `OptimizedTCPServer`
2. FIP `tab1` 处理后 -> `tab2`
3. `tab2` 输出 `FIPAlarmEvent`
4. DAS client -> `DASTcpServer`
5. DAS 原始包 -> `DASParseWorker`
6. `DASParseWorker` -> `DASPlotWorker`
7. FIP 与 DAS 包 -> `AlignedSessionCoordinator`
8. `AlignedSessionCoordinator` -> `JointRawStorageWorker`
9. `tab2` 事件 -> `Tab4LocalizationManager`
10. `Tab4LocalizationManager` 从 `AlignedSessionCoordinator` 拉取触发时间窗数据
11. `LocalizationWorker` 输出排序结果
12. `LocalizationStorageWorker` 写 NPZ/CSV

### 5.2 Tab3 数据流

- DAS TCP 收包
- 按包头解析
- 恢复为二维矩阵 `shape=(channel_count, samples_per_channel)` 或等价表达
- 送入实时绘图线程
- 与 FIP 包按 `comm_count` 对齐
- 写入联合原始缓存
- 按配置定时写 `FIP + DAS` 原始联合数据

### 5.3 Tab4 数据流

- 接收 `tab2` 输出的 `FIPAlarmEvent`
- 根据事件起止时间和前后窗口，从对齐缓存中提取数据
- 缺失包补零
- 对 DAS 数据做按通道特征提取
- 排除指定通道
- 形成候选定位结果
- 输出到 UI 和存储层

## 6. 线程与模块设计

### 6.1 Tab3 建议线程

#### `DASTcpServer`

职责：

- 监听 DAS 端口
- 接收完整包
- 解析包头
- 校验包长度
- 发出 `DASRawPacket`

不负责：

- 绘图
- 特征计算
- 存储

#### `DASPlotWorker`

职责：

- 从 DAS 包生成 `space-time` 绘图数据
- 生成曲线 1 / 曲线 2 所需数据
- 根据通道号提取单通道时域信号
- 应用轻量显示用滤波和降采样

#### `AlignedSessionCoordinator`

职责：

- 会话管理
- FIP/DAS 双源包按 `comm_count` 对齐
- 缓存最近 `10s` 对齐数据
- 缺失包状态维护
- 提供按时间窗或编号范围的数据提取接口

说明：

- 该模块是 `tab3/tab4` 的核心共享层。
- `tab3` 不自行维护一份对齐逻辑。
- `tab4` 也不自行维护一份对齐逻辑。

#### `JointRawStorageWorker`

职责：

- 以定时方式保存对齐后的原始联合数据
- 文件内容包括：
  - FIP 原始数据
  - FIP 特征
  - DAS 原始矩阵
  - 会话元数据
  - 缺失包信息

### 6.2 Tab4 建议线程

#### `DASTab4FeatureWorker`

职责：

- 对触发时间窗内的 DAS 数据按通道计算特征
- 支持最多 4 种特征
- 阈值基线按通道自适应
- 排除指定通道

#### `LocalizationWorker`

职责：

- 根据 FIP 事件和 DAS 特征结果进行定位排序
- 算法独立为单独文件，便于后续替换

排序分数定义：

- `score = duration_match + peak_strength + threshold_window_ratio`

说明：

- 分数由策略对象或配置项管理，不写死在 UI 层。
- 后续若算法升级，只替换该模块。

#### `LocalizationStorageWorker`

职责：

- 写事件触发 NPZ
- 写定位结果 CSV
- 写缺失包信息

## 11. 与现有代码的接入建议

### 11.1 保持现有 Tab1/Tab2 架构不变

当前工程中：

- `tab1` 已经是独立线程处理链
- `tab2` 已经是独立 worker pipeline

因此新增代码不要回写进 `tab1` 内部线程逻辑，只做最小接入：

- 在 `main.py` 中增加 `DASTab3Manager`
- 在 `main.py` 中增加 `AlignedSessionCoordinator`
- 在 `main.py` 中增加 `Tab4LocalizationManager`
- 将 `tab2` 的 `alarm_event_ready` 连接到 `Tab4LocalizationManager`

### 11.2 FIP 对齐输入建议

用于对齐和联合存储的 FIP 数据建议使用：

- `ProcessedData.unwrapped_data`
- `ProcessedData.downsampled_data`
- `ProcessedData.comm_count`
- `packet_duration_seconds = 0.2`

理由：

- `tab3`/`tab4` 使用统一的 `comm_count` 时间轴即可。
- 不需要直接复用 `tab1` 的绘图控件或绘图缓冲。

## 12. 开发时需要额外注意的问题

### 12.1 高通道数压力

- `space-time` 图在高通道数下很容易成为性能瓶颈。
- 必须优先做显示降采样，而不是把全矩阵直接喂给 UI。

### 12.2 补零与算法偏差

- 补零虽然能保证流程不断，但会改变短时能量等特征分布。
- 所以定位结果必须带上完整性标签，避免误判为高可信结果。

### 12.3 FIP 与 DAS 采样率不同

- 两侧仅按包编号对齐，不能直接按样本点一一对齐。
- 事件时间窗提取时，要分别按各自采样率重建本侧时间轴。

### 12.4 客户端尚未开发

- `tab3` 开发阶段必须同时准备一个 DAS 模拟发送器。
- 否则服务端功能只能停留在静态代码级验证。

### 12.5 UI 启停策略

- FIP 服务端和 DAS 服务端独立启动。
- 不建议继续沿用“单个 tab1 按钮控制全系统”。
- 后续可选增加主窗口级“全部开始/全部停止”。
